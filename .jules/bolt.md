## 2024-05-24 - [Avoid `len(query.all()) > 0` for SQLModel existence checks]
**Learning:** Checking for existence using `len(query.all()) > 0` in SQLModel/SQLAlchemy will fully execute the query and load all matching models into memory, leading to an N+1 memory footprint. Also learned from codebase context that calling `.first()` on the result object without `.limit(1)` in this setup fails to prevent full database queries.
**Action:** Always append `.limit(1)` to the `select()` statement before execution (e.g., `db.exec(select(...).limit(1)).first() is not None`) to ensure the DB limits the scan natively.
## 2024-05-24 - [Batch database inserts to avoid N+1 transaction overhead]
**Learning:** Found multiple instances where `db.add()`, `db.commit()`, and `db.refresh()` were called inside loops for inserting related rows. This leads to a separate database transaction per iteration, significantly increasing I/O overhead and transaction latency.
**Action:** When inserting multiple rows dynamically within a loop, collect the ORM objects into an array and execute a single `db.add_all(array)` followed by one `db.commit()` outside the loop. Only use `db.flush()` inside the loop if intermediate object IDs are strictly required before commit. Also, avoid `db.refresh()` unless the inserted object's attributes are explicitly accessed later in the same transaction context to further reduce database SELECT calls.
